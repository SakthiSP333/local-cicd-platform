1. The Big Picture

1.2 End-to-end flow (timeline / sequence view)

1. Developer runs git push against go-api on GitHub.
2. Jenkins' seed job go-api-pipeline polls the repo every ~5 minutes (SCMTrigger('H/5 * * * *')) and detects the new commit.
3. Pipeline runs go mod download → lint/format → static security scans → unit tests + benchmarks in parallel.
4. Pipeline builds a multi-stage Docker image tagged with the Jenkins BUILD_NUMBER.
5. Trivy scans the image; pipeline fails the build on any HIGH/CRITICAL CVE.
6. Pipeline logs into Harbor with the harbor-robot credential and pushes :BUILD_NUMBER and :latest.
7. Pipeline clones go-api-helm over SSH, runs update-chart.sh to bump values.yaml's image.tag and Chart.yaml's version, commits [skip ci], and pushes to main.
8. ArgoCD reconciles go-api-helm on its default ~3-minute polling interval, diffs live cluster state against the new Git commit.
9. ArgoCD applies the delta (new image tag) — automated: {prune: true, selfHeal: true} means it self-corrects any manual drift too.
10. Kubernetes pulls the new image from Harbor (via host.minikube.internal:8082 from inside the cluster), runs startup/liveness/readiness probes, and performs a zero-downtime rolling update.
11. The app's /metrics endpoint is scraped by Prometheus and visualized live in the pre-provisioned Grafana dashboard.
Why two different Git protocols? ArgoCD reads go-api-helm over plain HTTPS (public repo, read-only, no credentials needed). Jenkins writes to the same repo over SSH (needs push access). This asymmetry is a deliberate least-privilege split: the thing that only reads never holds write credentials.

2. Docker

2.1 Core concepts

TermMeaning
ImageImmutable, layered filesystem + metadata used to spin up containers. Built from a Dockerfile.
ContainerA running (or stopped) instance of an image — an isolated process with its own filesystem, network namespace, and cgroup limits.
LayerEach Dockerfile instruction (RUN, COPY, ...) adds a cached, reusable filesystem layer.
Multi-stage buildMultiple FROM blocks in one Dockerfile; later stages copy only the artifacts they need from earlier ones, so build tools never ship in the final image.
RegistryA server that stores and serves images by name:tag (Docker Hub, Harbor, ECR, GHCR...).
Docker-outside-of-Docker (DooD)Mounting the host's /var/run/docker.sock into a container so that container's docker CLI drives the host's daemon, instead of running a nested daemon (Docker-in-Docker).

2.2 In this project: go-api/Dockerfile (production, multi-stage)

# Stage 1: compile a static binary
FROM golang:1.21-alpine3.18 AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY cmd/ internal/ pkg/ ./
RUN CGO_ENABLED=0 GOOS=linux GOARCH=amd64 \
    go build -ldflags="-w -s" -o /app/go-api ./cmd/api/main.go

# Stage 2: tiny runtime image, non-root user
FROM alpine:3.18.4
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
COPY --from=builder /app/go-api /app/go-api
USER appuser
EXPOSE 8080
HEALTHCHECK CMD wget --spider http://localhost:${PORT}/healthz || exit 1
ENTRYPOINT ["/app/go-api"]

Why this matters: stage 1's Go toolchain (~350 MB) never reaches the final image — only the compiled, stripped (-ldflags="-w -s") static binary does. CGO_ENABLED=0 means no libc dependency, so a minimal alpine base is enough. Running as a non-root appuser and dropping capabilities (enforced again later at the Kubernetes securityContext level) limits blast radius if the container is ever compromised.

2.3 Dev image: Dockerfile.dev + docker-compose.yml

A separate, single-stage image installs air (hot-reload) and mounts source code as a bind volume, so local edits recompile instantly without rebuilding the image — never used in the CI path, only for local development.

3. Kubernetes

3.1 Core objects used in this project

Pod

Smallest deployable unit — one or more containers sharing network/storage. You rarely create Pods directly; a Deployment manages them.

Deployment

Declares desired replica count & Pod template; drives RollingUpdate to roll out new versions with zero downtime.

Service

Stable virtual IP + DNS name in front of a set of Pods, selected by label. This project uses ClusterIP (internal only).

Ingress

HTTP(S) routing rule mapping a hostname/path to a Service, handled by the ingress-nginx controller (a Minikube addon here).

ConfigMap

Non-secret key/value config injected as env vars or files. Holds ENV, LOG_LEVEL, PORT here.

Secret

Same idea as ConfigMap but for sensitive data (base64-encoded, not encrypted by default). Holds AUTH_TOKEN here.

Namespace

Logical partition of a cluster for multi-tenancy — this project uses one per environment (go-api-dev/stage/prod).

ServiceAccount / RBAC

Identity a Pod runs as, and Role/RoleBinding (namespace-scoped) or ClusterRole/ClusterRoleBinding (cluster-scoped) grants for that identity.

HPA (HorizontalPodAutoscaler)

Automatically scales replica count based on observed metrics (CPU % here).

PDB (PodDisruptionBudget)

Guarantees a minimum number of Pods stay up during voluntary disruptions (node drains, cluster upgrades).

NetworkPolicy

Firewall rules for Pod-to-Pod / Pod-to-external traffic, enforced by the CNI.

PVC (PersistentVolumeClaim)

A request for durable storage, bound to a PersistentVolume by a StorageClass.

3.2 Probes — how Kubernetes knows a Pod is healthy

ProbePurposeIn this chart
startupProbeGives slow-starting apps grace time before liveness kicks inGET /livez, every 2s, 10 tries
livenessProbeRestarts the container if it stops respondingGET /livez, every 10s
readinessProbeRemoves the Pod from Service endpoints if not ready to serve traffic (without restarting it)GET /readyz, every 5s

3.3 Deployment manifest, annotated (go-api-helm/templates/deployment.yaml)

spec:
  strategy:
    type: RollingUpdate       # replace pods gradually, never all-at-once
    rollingUpdate:
      maxSurge: 25%             # up to 25% extra pods during rollout
      maxUnavailable: 25%       # at most 25% of pods down at once
  template:
    spec:
      securityContext:          # pod-level
        fsGroup: 2000
      containers:
        - securityContext:       # container-level, more restrictive
            capabilities: { drop: [ALL] }
            runAsNonRoot: true
            runAsUser: 10001
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
          env:
            - name: AUTH_TOKEN
              valueFrom: { secretKeyRef: { name: go-api-secret, key: AUTH_TOKEN } }
          startupProbe:   { httpGet: { path: /livez,  port: http } }
          livenessProbe:  { httpGet: { path: /livez,  port: http } }
          readinessProbe: { httpGet: { path: /readyz, port: http } }

3.4 Networking model in this project

The chart's Service is ClusterIP (port 80 → container port 8080). Ingress (nginx class) is only enabled by default in dev; staging/prod also enable it with environment-specific hostnames (go-api.stage.local, go-api.prod.local, TLS redirect on in prod). Locally, make port-forward exposes each environment's Service directly on the host (dev 8081, stage 8083, prod 8084) — bypassing Ingress entirely for quick access.

4. Helm

4.1 Core concepts

TermMeaning
ChartA packaged bundle of Kubernetes templates + metadata (Chart.yaml) + default config (values.yaml). This project has one chart: go-api-helm.
Values fileYAML overrides fed into the chart's templates. Helm merges multiple -f files left-to-right, later files win.
TemplateA Kubernetes manifest with Go-template placeholders ({{ .Values.x }}), rendered at install/upgrade time.
ReleaseA named, versioned instance of a chart installed into a cluster/namespace (e.g. go-api-dev).
Helper template (_helpers.tpl)Reusable named templates (e.g. go-api.fullname, go-api.labels) shared across manifests to avoid repetition.

4.2 Environment overlay pattern used here

Rather than one values file per cluster, this chart layers a shared base over an environment-specific overlay — exactly what ArgoCD's helm.valueFiles list does:

helm.valueFiles:
  - values.yaml        # shared defaults
  - values-dev.yaml     # environment-specific overrides, applied last (wins)

Learning note: values-prod.yaml hardcodes a mock authToken with a comment admitting: "In enterprise setups, this is injected via vault/external secrets." This repo intentionally does not implement Vault/External-Secrets — a good next exercise once you're comfortable with the basics.

5. Harbor

5.1 Core concepts

TermMeaning
RegistryHarbor itself — an OCI-compliant image registry with a web UI, RBAC, and vulnerability scanning built in (an open-source alternative to Docker Hub / ECR / GCR).
ProjectA namespace inside Harbor grouping repositories, with its own access policy (public/private) and quotas. This project uses one project: library.
Robot accountA non-human, scoped credential (e.g. robot$jenkins-robot) used by CI systems instead of a real user's password.
Trivy adapterAn optional Harbor component giving the registry its own built-in CVE scanner UI, independent of any scanner run by CI.

5.2 How this project's Harbor is configured

Installed by scripts/install-harbor.sh (Harbor v2.8.2, official online installer), then patched via sed into:

hostname: localhost
http:
  port: 8082              # moved off the default 80 to avoid clashing with Jenkins on 8080
harbor_admin_password: Harbor12345
# https: block fully commented out -> HTTP-only registry
Why HTTP-only matters: container runtimes refuse to pull from a plain-HTTP registry by default ("insecure registry"). That's why Minikube in this project is started with --insecure-registry="host.minikube.internal:8082" — explicitly whitelisting Harbor as an exception. In production you would terminate TLS instead.

5.3 Dual addressing — one registry, two hostnames

CallerAddress usedWhy
Jenkins (host Docker daemon)localhost:8082Jenkins' Docker CLI talks to the host daemon directly (Docker-outside-of-Docker)
Kubelet inside Minikubehost.minikube.internal:8082From inside the Minikube VM, localhost means the VM itself, not your laptop — Minikube provides this DNS alias to reach the host

5.4 The robot account (scripts/create-harbor-project.sh)

Automated purely via Harbor's REST API (http://localhost:8082/api/v2.0): creates the public library project, then a system-level robot jenkins-robot scoped to repository:push, repository:pull, artifact:create, artifact:delete on that project only — never full-admin access. Credentials land in infrastructure/harbor-robot.env (gitignored, chmod 600):

HARBOR_REGISTRY=localhost:8082
HARBOR_PROJECT=library
HARBOR_ROBOT_USER=robot$jenkins-robot
HARBOR_ROBOT_SECRET=<generated>
Gotcha: the robot username literally contains a $ character. Anything that later sources this env file must quote the variable — unquoted shell expansion would try to interpret $jenkins-robot as a variable reference.

6. Jenkins

6.1 Core concepts

TermMeaning
Declarative PipelineA Jenkinsfile written in a structured pipeline { stages { ... } } DSL, checked into the app repo (pipeline-as-code).
StageA named phase of the pipeline (Lint, Test, Build, Scan, Push...), shown as a step in Jenkins' UI ("Blue Ocean" style graph).
AgentWhere a stage/pipeline actually executes. This project uses agent any — run directly on the single Jenkins controller container (no separate build agents).
Credentials storeJenkins' encrypted vault for secrets (SSH keys, username/password pairs), referenced by ID in the Jenkinsfile — never hardcoded.
SCM polling triggerJenkins periodically checks the source repo for new commits (vs. a webhook, which pushes notifications instantly). This project polls every ~5 min via cron syntax H/5 * * * *.
init.groovy.dGroovy scripts a Jenkins controller auto-runs on JVM startup, before the setup wizard — used here for fully headless (no-click) provisioning.

6.2 Headless provisioning (scripts/install-jenkins.sh)

Rather than clicking through Jenkins' setup wizard, three Groovy scripts are dropped into jenkins_home/init.groovy.d/ and run automatically on boot:

ScriptDoes
00-security.groovyCreates the admin user, enables a private security realm, disables anonymous read access
10-credentials.groovyRegisters github-personal-ssh (SSH key) and harbor-robot (username/password) into the global credentials store
20-seed-job.groovyProgrammatically creates the go-api-pipeline job pointed at your fork, with the SCM poll trigger — and explicitly starts that trigger
Real bug fixed in this repo's git history: the seed job originally built its SCMTrigger object but never called trigger.start(job, true) before job.save() — so the trigger existed in config but never actually scheduled polls. The fix (commit "Fix seed pipeline job's SCM poll trigger not activating") added that missing call. A good reminder that configuring an object isn't the same as activating it.

The Jenkins image itself (infrastructure/jenkins/Dockerfile) is jenkins/jenkins:lts-jdk17 plus the Docker CLI (bind-mounted host socket = Docker-outside-of-Docker), Go, and the security toolchain (gosec, trivy, golangci-lint, govulncheck, gitleaks) baked in so no stage needs to install tools on the fly.

6.3 The pipeline, stage by stage (go-api/Jenkinsfile)

pipeline {
  agent any
  environment {
    REGISTRY   = "localhost:8082"
    IMAGE_NAME = "library/go-api"
  }
  stages {
    stage('Lint & Format')          // gofmt, go vet, golangci-lint
    stage('Static Security Analysis') // gosec, govulncheck, gitleaks
    stage('Testing') { parallel {  // unit tests + benchmarks concurrently
        stage('Unit Tests')  { sh "go test -v -race -coverprofile=coverage.out ./..." }
        stage('Benchmarks') { sh "go test -bench=. -run=^$ ./..." }
    } }
    stage('Docker Build')   // docker build over the DooD socket
    stage('Trivy Image Scan') // exit-code 1 on HIGH/CRITICAL CVE = fails build
    stage('Push to Harbor')  // docker login + push :BUILD_NUMBER and :latest
    stage('Update GitOps Chart') { sshagent(['github-personal-ssh']) {
        // clone go-api-helm, bump image.tag via update-chart.sh, commit, push
    } }
  }
}

Quality gates that actually fail the build: unformatted code (gofmt -l . non-empty), go vet/golangci-lint issues, and any HIGH/CRITICAL Trivy finding. Non-blocking: gitleaks detect failures are caught with || echo ... — intentionally advisory only in this sandbox.

7. ArgoCD

7.1 Core concepts — what "GitOps" actually means

GitOps: Git is the single source of truth for desired cluster state. Instead of a CI job running kubectl apply imperatively, a controller (ArgoCD) continuously pulls the desired state from Git and reconciles the live cluster to match it — including reverting manual kubectl changes ("self-heal") if drift is detected.

TermMeaning
Application (CRD)ArgoCD's core object: binds a Git source (repo/path/revision) to a cluster destination (server/namespace).
SyncThe act of reconciling live state to match Git. Can be manual (click "Sync") or automated.
PruneWhen automated, delete cluster resources that were removed from Git.
Self-healWhen automated, revert any manual/out-of-band change back to what Git declares.
ApplicationSetA generator that stamps out multiple Applications from one template (e.g. one per environment).
App-of-AppsA common pattern (not used in this project) where one root Application manages a set of child Applications.

7.2 This project's Application (go-api-helm/argocd/application.yaml)

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: go-api-dev
  namespace: argocd
spec:
  source:
    repoURL: https://github.com/SakthiSP333/go-api-helm.git  # HTTPS, read-only
    targetRevision: main
    helm:
      valueFiles: [values.yaml, values-dev.yaml]
  destination:
    server: https://kubernetes.default.svc
    namespace: go-api-dev
  syncPolicy:
    automated: { prune: true, selfHeal: true }
    syncOptions: [CreateNamespace=true]
Only apply one of the two ArgoCD manifests. application.yaml (single dev Application, applied by install-argocd.sh) and application-set.yaml (fans out dev/stage/prod via a list generator) both try to own an Application named go-api-dev — applying both causes a conflict. Pick one strategy per cluster.

7.3 Install mechanism used here

Unlike the monitoring stack, ArgoCD is installed from the official upstream manifest rather than a Helm chart:

kubectl create namespace argocd
kubectl apply -n argocd -f \
  https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

Admin credentials come from the auto-generated argocd-initial-admin-secret Kubernetes Secret — retrieved once at install time and saved to infrastructure/argocd-creds.env.

8. Prometheus & Grafana

8.1 Core concepts

TermMeaning
PrometheusPull-based metrics database: it scrapes HTTP /metrics endpoints on a schedule and stores time-series data.
CounterA metric that only goes up (e.g. total requests). Use rate() in queries to get a per-second rate.
HistogramBuckets observations (e.g. request duration) so you can compute quantiles like p95/p99 after the fact.
ServiceMonitor (CRD)Prometheus-Operator's declarative "scrape this Service" config — the Kubernetes-native way to tell Prometheus what to scrape, instead of hand-editing prometheus.yml.
PromQLPrometheus' query language, used both for alerting rules and Grafana panels.
GrafanaVisualization layer on top of Prometheus (and other data sources) — dashboards, panels, alerting.

8.2 Metrics instrumentation in the app (go-api/internal/middleware/metrics.go)

http_requests_total           // CounterVec, labels: method, path, status
http_request_duration_seconds // HistogramVec, default buckets

// wired into Gin as middleware, exposed unauthenticated at GET /metrics:
router.GET("/metrics", gin.WrapH(promhttp.Handler()))

Note /metrics sits outside the /api/v1 auth group — Prometheus scrapes it with no bearer token, matching the convention that metrics endpoints are typically network-isolated rather than app-authenticated.

8.5 Laptop-friendly resource tuning

install-monitoring.sh installs the kube-prometheus-stack Helm chart with persistence disabled, a short 6h retention window, tiny CPU/memory requests across every component, and Alertmanager disabled entirely — this is a learning sandbox, not a durable observability stack.

11. Troubleshooting Playbook

Straight from this project's README — the four failure modes it explicitly documents:

SymptomRoot causeFix
Kubernetes can't pull images from Harbor ("insecure registry" error)Container runtimes block plain-HTTP registries by defaultConfirm Minikube was started with --insecure-registry="host.minikube.internal:8082" (minikube profile list); if missing, make clean && make setup

11.1 General debugging order-of-operations

1. kubectl get pods -n <ns> — is it Running, CrashLoopBackOff, Pending, ImagePullBackOff?
2. kubectl describe pod <pod> -n <ns> — check the Events section at the bottom first.
3. kubectl logs <pod> -n <ns> (add --previous if it already restarted).
4. For scheduling issues, check node resources: kubectl top nodes, kubectl describe node.
5. For CI issues, read the failing stage's console output in Jenkins top-to-bottom — declarative pipelines stop at first failure.
6. For GitOps drift, argocd app diff before assuming ArgoCD "isn't working" — it may be correctly reporting a real difference.
